Immigroov — Booking System Workflow

How the booking system actually works today — every node maps to a live database function, trigger or cron job.

AS-BUILT · verified 25 Jun 2026 Money is mock — recorded in booking_ledger
A booking is created and confirmed immediately when a customer pays for a single session (there is no separate mentor "accept" step, and no packages yet). From confirmed it can move through three paths — cancel, reschedule, or no-show. Each branch is gated by the deadline state of the session: booking_deadline_state(slot) returns free (≥24h before), late (2–24h before, needs approval or incurs a penalty), or buffer (under 2h before — self-service is blocked). The approval window is response_window = MIN(now+48h, slot−2h). All payments are mock, so every refund / credit / penalty is written as a row in booking_ledger rather than actually charged.

Diagram 3 — Reschedule flow
Max 2 reschedules per booking. A 3rd attempt by either party → force_autocancel: booking cancelled, customer refund 100%, 100% penalty on whoever initiated. The under-2h buffer is enforced only on customer-initiated reschedule (customer_reschedule / request_reschedule); the mentor's propose and the mentee's accept/ask-date are not buffer-gated in code (though an offer made under 2h has its window in the past and is expired by the cron almost immediately).

An unconfirmed session is not auto-cancelled — the attend-check is only a nudge; it simply auto-completes after the end time unless a no-show is reported first.
